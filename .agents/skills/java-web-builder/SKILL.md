# Java Web 构建器

## 目标

将 Java Web 功能从想法落地到可验证实现，并在每个阶段产出清晰结果。

## 强制流程门禁

- 如果项目未初始化，先提示用户初始化项目，默认按照 [examples/](examples/) 目录下的示例进行初始化。
- 如遇到不清晰或模糊的要求执行内容，必须先与用户澄清，得到明确要求后才能进入下一步。
- 每一步完成后，必须先与用户确认，得到明确同意后才能进入下一步。

## 工作流程

### 1. 需求分析

- 和用户讨论，明确需求。
- 如果需求太大，需拆成多个独立的小需求。
- 将模糊需求转化为可执行、可验收的标准。
- 产出或更新需求文档。

### 2. 详细设计

- 根据第一步需求分析文档进行详细设计，一个需求文档对应一个设计文档。
- 接口定义，数据流转，功能实现流程。
- 给出3种设计方案，供用户选择。
- 确定方案后产出或更新设计文档。

### 3. 代码编写

- 根据详细设计文档编写代码和注释。
- 保持与仓库现有风格一致，避免无关重构。

### 4. 代码检查

- 系统检查代码，排查潜在问题。
- 保持代码行为一致，避免引入功能回归。
- 如有问题，与用户确认后重新修改指定内容。

### 5. 测试验证

- 编写测试代码。
- 运行测试代码，通过测试。

### 6. 更新文档

- 必须更新 README、RECORDS。
- RECORDS 仅允许追加，不允许覆盖或删除历史记录。
- 文档保持简洁，并与最终实现一致。

### 7. 验收

- 检查需求文档，设计文档，代码，测试，README.md，RECORDS.md，确保所有内容一致且符合要求。
- 如果未达到验收标准，与用户确认后重新修改指定内容，直到达到验收标准。

## 执行规则

- 使用中文。
- 文件编码使用UTF-8，换行符使用LF。
- 需求不完整时，先明确假设再推进实现。
- 以当前仓库中的真实代码与配置为准，避免主观猜测。
- 保持改动最小化、可回滚。
- 在交付前完成验证；无法验证的部分必须明确说明。
- 编写代码和文档时参考 [examples/](examples/) 目录下的示例。
